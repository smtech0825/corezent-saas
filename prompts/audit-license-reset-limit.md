# audit-license-reset-limit

You are a senior Next.js 15 (App Router) / TypeScript / Supabase engineer auditing the **CoreZent** server codebase — the backend that the GenieWork desktop app calls for license validation and PC change (`/api/license/*`). This is a **READ-ONLY 점검**. 사실만 파일:줄 근거로 보고하고, 코드는 한 줄도 고치지 않는다.

## 점검 목적 (Steve의 질문)
GenieWork 앱에는 "PC변경" 버튼이 있다. 누르면 현재 PC가 미인증되고 새 PC를 인증할 수 있다(서버 `reset` 호출로 추정). Steve의 핵심 우려:
1. **PC변경(reset)을 반복하면 한도(예: 3대)보다 더 많은 PC에서 사용할 수 있는가?** (동시 사용은 3대로 묶이더라도, 유효기간 동안 거쳐가는 PC 총수가 무제한이 되는가)
2. reset이 슬롯을 **정확히 정리**하는가, 아니면 **옛 HWID 슬롯이 남아 한도가 새는가**.
3. reset 호출에 **인증/소유권/횟수 제한이 있는가**(키만 알면 누구나 무한 reset 가능한가).

배경(이미 확인된 사실, 근거 재확인 대상):
- GenieWork는 멀티 슬롯 구조. `license_keys.tier`("3pc"→3)를 숫자로 읽어 `hwid_mapping`에 키당 HWID 행을 쌓고, 한도 도달 시 차단(`_lib_supabase.ts` 추정 :91-99 HWID_LIMITS, :239-241 한도 비교, :245-249 insert).
- reset은 전체 삭제(올-오어-나싱)로 보임(`_lib_supabase.ts` 추정 :264-279), 소유권 대조 없음(`reset/route.ts:11` 주석).
- ※ 위 줄번호는 출발점일 뿐. 반드시 실제 코드를 열어 확인.

## STEP 0 — reset/재인증 파이프라인 지도
입력 → 라우트 → 데이터 변경 → 재인증까지 한 흐름으로 그린다:
- 앱 "PC변경"이 호출하는 서버 라우트: `src/app/api/license/reset/route.ts`(+ `_lib_supabase.ts`의 reset 함수). geniework 분기 경로 확인.
- reset이 `hwid_mapping`(또는 해당 테이블)에서 **무엇을 지우는지**(전체 행 삭제 vs 특정 HWID 1건).
- reset 직후 다음 validate에서 **새 HWID가 어떻게 다시 등록**되는지.
- ★ **'한도 우회 실배선' 점검**: reset→재등록 사이클을 반복하면 한도 카운트가 **실제로 리셋되어 무한 등록 가능한지**, 아니면 어딘가에 **누적 기록(거쳐간 총 PC 수)** 이 남아 막히는지.

## 점검 항목 (각 항목 파일:줄 근거)

### A. reset이 정확히 무엇을 지우나
- reset 핸들러가 `hwid_mapping`(키당 HWID 행)에서 **전체 삭제**인지 **특정 HWID 1건 삭제**인지. (앱이 보내는 파라미터에 특정 HWID가 있는지/없는지도 확인)
- reset 후 슬롯 카운트가 **0으로 떨어지는지**(= 다음에 다시 한도만큼 등록 가능).

### B. ★ 한도 우회 가능성 (핵심)
- reset → 새 HWID 등록 → reset → … 반복 시, **한도(3)를 넘는 누적 PC가 가능한지.** 즉 "동시 3대"는 지키지만 "기간 내 거쳐간 PC 총수"는 무제한인지.
- 어딘가에 **누적 활성화 횟수/거쳐간 HWID 이력**을 기록하고 그걸로 막는 로직이 있는지(예: 평생 활성화 횟수 상한). 없으면 "무제한 순환" 확정.

### C. reset 호출 통제 (인증·소유권·횟수)
- reset 라우트가 **키 값만으로** 실행되는가, 아니면 **로그인 사용자(소유자) 확인**이 있는가.
- reset에 **횟수/주기 제한**(예: 30일 1회)이 있는가. BotID·rate limit이 이 라우트에 걸려 있는가(PROJECT_STRUCTURE 기준 `/api/license/*`는 BotID 미적용).
- 키만 알면 **제3자도 무한 reset** 가능한지.

### D. reset과 한도 강제의 정합
- validate의 한도 비교(`HWID_LIMITS[tier]` vs 현재 등록 수)와 reset의 삭제가 **같은 테이블·같은 키 기준**으로 정확히 맞물리는가. 불일치(예: 일부만 지워 카운트가 꼬임)가 가능한가.
- 부분 실패(reset 도중 오류)로 **슬롯이 어중간하게 남는** 경로가 있는가.

### E. 부가 위험 (이미 의심된 것 재확인)
- **HWID 불안정**: 같은 PC가 네트워크/호스트명 변경으로 다른 HWID가 되면, 사용자가 reset을 더 자주 쓰게 되고(슬롯 낭비) → 그 자체가 우회 빈도를 높이는가.
- **TOCTOU**: 서로 다른 새 HWID 두 요청 동시 도착 시 둘 다 카운트 통과 후 둘 다 insert로 일시적 한도 초과(4번째) 가능한지(`_lib_supabase.ts` 추정 :231/:240 카운트 후 insert 사이 경합). DB측 UNIQUE/카운트 제약이 repo에 있는지.

## 반드시 답할 결론
1. **PC변경(reset) 반복으로 한도보다 많은 PC를 거쳐 쓸 수 있는가?** (예/아니오 + 근거 파일:줄)
2. reset은 **전체 삭제인가 개별 삭제인가**, 슬롯 정리가 정확한가.
3. reset에 **소유권/횟수 통제가 있는가**(없으면 키 공유 시 사실상 무제한인지).
4. (있다면) **한도가 새는 구체 경로** 1~N개를 파일:줄로.
