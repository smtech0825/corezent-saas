# fix-product-option-selector

You are a senior Next.js 15 (App Router) / TypeScript / Supabase / Tailwind v4 engineer on the **CoreZent** web codebase (멀티 상품 SaaS 스토어, 결제=Lemon Squeezy). 공개 사이트의 상품 카드에 **옵션 선택 UI**(주기·PC수 등)를 만들고, 선택 조합에 맞는 상품의 **Checkout URL로 결제**가 이어지게 한다. **사전 점검(READ-ONLY audit) 불필요 — 바로 작업.** 단, 작업 중 애매하거나 위험한 판단이 필요하면 **멈추고 한국어로 보고**한다.

## ★ 확정 설계 (이대로 — 재설계 금지)
- **DB 상품은 지금처럼 "조합별 개별 등록" 유지.** 예: GenieWork는 1/3/5/10PC × 월간/연간 = 최대 8개 상품 행, 각자 slug(`geniework_1pc_monthly` 등)·lemon_squeezy_variant_id·checkout_url 보유. **slug 규칙({family}_{tier}_{interval})과 결제→라이선스 파이프라인(slug 기반 분기)은 절대 불변.**
- 이 위에 **표시(진열) 계층만** 추가한다:
  1. **묶음 키(option group)**: 같은 카드로 묶일 상품들을 지정 (예: geniework 계열 8개 → 카드 1개). slug의 family 토큰에서 파생 가능하면 그걸 기본으로 쓰되, 관리자가 명시 지정할 수 있는 컬럼도 허용(둘 중 단순·명확한 쪽, 판단 후 보고).
  2. **옵션 라벨(관리자 자유 입력)**: 각 상품에 옵션 축 라벨을 입력 — 예: 축1(주기)=`월간`, 축2(수량)=`3PC용`. **라벨 텍스트는 완전 자유 입력**(하드코딩된 enum 금지 — 새 제품이 다른 옵션 개념을 쓸 수 있음). 축은 최대 2개면 충분(축1·축2), 1개만 쓰는 상품도 허용.
- **공개 카드 동작**: 묶음 키가 같은 상품들을 카드 1개로 렌더 → 옵션 축별로 **드롭다운(select)** 생성(등록된 라벨에서 자동, 버튼 나열 아님) → 선택 조합에 해당하는 상품의 **가격을 즉시 갱신 표시**(옵션마다 가격 다름) + 그 상품의 checkout_url로 구매 버튼 연결. 등록 안 된 조합(예: 5PC 연간 없음)은 드롭다운에서 비활성/제외. 기본 선택은 첫 유효 조합.
- **수량 선택 추가**: 카드에 수량 선택(1~N, 기본 1, 드롭다운 또는 스테퍼). 표시 가격 = 조합 가격 × 수량으로 즉시 갱신. 결제 연결은 checkout_url에 LS의 수량 파라미터를 붙여 전달(LS checkout link가 URL로 quantity를 받는지 확인 — 방식 확인 후 적용, 불가하면 멈추고 보고). ⚠️ **수량 N 결제 시 웹훅이 라이선스를 N개 발급하는지 확인**하고, 현재 웹훅이 수량을 무시한다면(1개만 발급) 그 사실을 보고 — 라이선스 N개 발급 로직 변경은 결제·라이선스 핵심이므로 승인 후 별도.
- **관리자 제품 등록/편집**: 묶음 키·옵션 라벨(축1/축2) 입력 필드 추가. 기존 Variant ID·Checkout URL 필드는 그대로(조합별 상품에 각각 입력하는 현 방식 유지).
- LS API 호출 없음 — 등록된 checkout_url로 연결만.

## 공통 STRICT RULES
- 코드는 네가 안다. 사용자는 코드 세부를 모르니, **막히거나 애매한 결정이 생기면 추측하지 말고 멈추고 한국어로 보고**하고 지시를 기다린다.
- **보편 해결 / 하드코딩 금지.** 옵션 라벨·묶음을 코드에 하드코딩하지 말 것(전부 DB에서). "geniework 전용" 분기 금지 — 어떤 제품군이든 같은 구조로 동작해야 함.
- **결제·라이선스 핵심 불변**: slug 규칙·variant_id 매칭·HMAC·멱등성·라이선스 발급 미접촉. 이 작업은 순수 표시 계층.
- **가격 표시**: 기존 `formatPrice`(lib/price.ts) 재사용.
- **다른 작업자/다른 CC 세션 미커밋 변경 미접촉.** `git add .` 금지(변경 파일만 개별 add). **Wave 중 push 금지**(최종에서만).
- **각 Wave 끝**: `npx tsc --noEmit` + `npm run build` 통과 → 변경 파일만 개별 커밋 → 멈추고 보고 → 승인 후 다음 Wave. **(bug-detective·code-guardian은 Wave마다 돌리지 말 것 — 최종에 한 번만.)**
- 각 보고: **①관찰 ②근거(파일:줄) ③공용 영향 범위 ④결정적 코드임.** 한국어로.
- 스키마 변경은 **마이그레이션 파일만 작성**(비파괴 ADD COLUMN·기본값 명시), 적용은 Steve가 SQL Editor에서 직접. 기존 상품 백필값(현 GenieWork 1PC 월간의 묶음 키·라벨)은 마이그레이션 주석에 예시 SQL로 제공(값 확정은 Steve).

## WAVE 1 — 스키마 + 관리자 입력 필드
- products(또는 적절한 테이블)에 표시 계층 컬럼 추가: 묶음 키(option_group 등), 옵션 축1 라벨, 축2 라벨 (+축 이름 표시용 컬럼이 필요하면: 예 "주기"/"PC 수" — 카드에서 축 제목으로 보여줄 텍스트, 이것도 자유 입력). 네이밍·최소 구성은 네가 판단하되 **축 라벨은 자유 텍스트** 원칙.
- 마이그레이션 파일 작성(비파괴). 기존 행 백필 예시 주석 포함.
- 관리자 제품 등록/편집 폼(ProductForm)에 해당 입력 필드 추가 — 기존 Variant ID·Checkout URL 옆에 자연스럽게. 필수 아님(옵션 미사용 상품은 비워두면 단독 카드로 동작).
검증 → 변경 파일만 커밋 → **STOP & 보고.**

## WAVE 2 — 공개 카드 옵션 선택 UI (드롭다운 + 수량 + 가격 갱신)
- 공개 상품 목록/랜딩에서 **묶음 키가 같은 상품들을 카드 1개로** 렌더. 옵션 축별 **드롭다운(select)**(등록된 라벨에서 자동 생성, 중복 제거·등록 순서 유지) → 선택 조합의 상품 **가격이 즉시 갱신**(formatPrice, 옵션마다 가격 다름) + 구매 버튼이 그 상품의 checkout_url로.
- **수량 선택**(1~N 드롭다운 또는 스테퍼, 기본 1): 표시 가격 = 조합 가격 × 수량 즉시 갱신. checkout_url에 LS 수량 파라미터를 붙여 전달 — LS checkout link의 quantity URL 파라미터 지원 방식 확인 후 적용, URL로 불가하면 멈추고 보고. 수량 상한은 합리적 기본(예: 10) — 과설계 금지.
- **웹훅 수량 처리 확인(보고 항목)**: 수량 N 결제 시 현재 웹훅이 라이선스를 N개 발급하는지 코드로 확인해 보고. 무시한다면(1개만 발급) 구현하지 말고 사실만 보고 — 발급 로직 변경은 승인 후 별도.
- 미등록 조합은 드롭다운에서 비활성/제외. 묶음 없는 상품(묶음 키 비어있음)은 기존처럼 단독 카드.
- 기본 선택: 첫 유효 조합(또는 가장 저렴한 조합 — 판단 후 보고).
- 디자인은 기존 카드 톤 유지. 모바일에서도 드롭다운·수량이 깨지지 않게.
- 상품 상세 페이지(`/product/[slug]`)가 이미 있으면: 상세에서도 같은 선택기 노출 여부 판단 후 적용(같은 컴포넌트 재사용).
검증 → 커밋 → **STOP & 보고.**

## 멈춤 규칙
- 묶음 키를 slug 파생 vs 명시 컬럼 중 뭘로 할지 판단이 서면 구현 전 근거와 함께 보고(둘 다 가능하면 단순한 쪽).
- LS checkout link가 URL로 quantity를 못 받으면(동적 checkout API 필요) 구현 전 멈추고 보고.
- 웹훅이 수량을 무시해 라이선스 N개 발급 로직 변경이 필요하면 구현하지 말고 보고(결제·라이선스 핵심).
- 기존 상품 카드 렌더 구조가 묶음 렌더와 충돌하면(큰 리팩터 필요) 멈추고 보고.
- slug/결제/라이선스 로직에 닿을 위험이 보이면 즉시 멈춰 보고.
