# fix-stage2-member

You are a senior Next.js 15 (App Router) / TypeScript / Supabase / Tailwind v4 engineer on the **CoreZent** web codebase (멀티 상품 SaaS 스토어). 2단계 "회원 기능" 3종을 Wave별로 처리한다. **사전 점검(READ-ONLY audit) 불필요 — 바로 작업.** 단, 작업 중 애매하거나 위험한 판단이 필요하면 **멈추고 한국어로 보고**한다.

## 공통 STRICT RULES
- 코드는 네가 안다. 사용자는 코드 세부를 모르니, **막히거나 애매한 결정이 생기면 추측하지 말고 멈추고 한국어로 보고**하고 지시를 기다린다.
- **보편 해결 / 하드코딩 금지.** in-place 수정, 지정 범위만, 무관 리팩터 금지.
- **결제·라이선스·웹훅 로직 불변**: 이번 3종은 회원 UI/설정 계열. HMAC·멱등성·variant_id 매칭·라이선스 발급 경로 미접촉.
- **GW_SUPABASE(외부 라이선스 DB)·GeniePost Sheets 미접촉** (이번 범위 아님).
- **다른 작업자/다른 CC 세션 미커밋 변경 미접촉.** `git add .` 금지(변경 파일만 개별 add). **Wave 중 push 금지**(최종에서만).
- **각 Wave 끝**: `npx tsc --noEmit` + `npm run build` 통과 → 변경 파일만 개별 커밋 → 멈추고 보고 → 승인 후 다음 Wave. **(bug-detective·code-guardian은 Wave마다 돌리지 말 것 — 최종에 한 번만.)**
- 각 보고: **①관찰 ②근거(파일:줄) ③공용 영향 범위 ④결정적 코드임.** 한국어로.
- 개인정보·계정 삭제 등 되돌릴 수 없는 동작은 **확인 절차(모달·재확인)** 필수. 조용히 삭제 금지.

## WAVE 1 — 내 라이선스에서 앱 설치파일 다운로드
- 회원 대시보드의 "내 라이선스"(또는 라이선스 상세) 화면에, 구매한 제품의 **앱 설치파일 다운로드 버튼**을 추가. 현재는 외부 구글사이트 링크로 안내됨 → 사이트 내에서 바로 받게.
- 다운로드 URL 출처: 기존에 changelog/릴리스 정보에 `download_urls`(또는 유사 필드)가 있으면 그것을 사용. 제품별·OS별(있으면) 구분. **URL 하드코딩 금지 — DB/기존 데이터에서 읽기.**
- 표시: 버전(있으면)·간단한 릴리스 노트 링크와 함께. 다운로드 대상이 없는 제품(웹 전용 등)이면 버튼 숨김.
- ⚠️ 설치파일 자체의 업로드/버전 관리(관리자 백엔드)는 이번 범위 아님 — **기존 데이터에 있는 URL로 링크만**. 데이터가 없거나 구조가 불명확하면 멈추고 보고.
검증 → 변경 파일만 커밋 → **STOP & 보고.**

## WAVE 2 — 회원 탈퇴 + 알림/마케팅 수신 동의
### 2-A. 회원 탈퇴 메뉴
- 회원 설정(계정 설정) 화면에 **회원 탈퇴** 기능 추가. 개인정보 자기결정권상 필요.
- **되돌릴 수 없는 동작**이므로: 확인 모달 + (가능하면) 재확인(비밀번호 확인 또는 "탈퇴" 문구 입력). 
- 탈퇴 처리 정책은 **기존 코드/스키마를 보고 안전한 방식** 선택 후 보고: Supabase Auth 사용자 삭제 vs soft-delete(비활성 플래그). ⚠️ **활성 구독·라이선스가 있는 회원의 탈퇴 시 처리**(구독 취소 안내·데이터 정리)가 얽히면, 삭제 로직을 임의로 정하지 말고 **멈추고 보고**. 
- 결제/구독 데이터(orders·subscriptions)와의 FK 관계를 깨지 않게(주문 이력 보존 필요 여부 판단 → 애매하면 보고).

### 2-B. 알림/마케팅 수신 동의 설정
- 설정 화면에 **알림/마케팅 이메일 수신 동의** 토글 추가. 저장은 profiles(또는 유사) 테이블의 boolean 컬럼.
- 컬럼이 없으면: **마이그레이션 파일 작성**(비파괴 ADD COLUMN, 기본값 지정). 적용은 Steve가 SQL Editor에서 직접(CC는 외부/DB 직접 적용 금지). 파일 상단에 대상 DB·기본값 명시.
- 실제 발송부(Nodemailer 등)에서 이 동의값을 존중하는 연결은, 발송 트리거가 이번 범위에 있으면 반영하고, 광범위하면 "설정 저장까지" 하고 발송부 연동은 보고 후 별도.
검증 → 커밋 → **STOP & 보고.**

> ⚠️ 2-A와 2-B는 한 화면(설정)에 들어가지만 성격이 달라 한 Wave로 묶되, 커밋은 논리적으로 분리(탈퇴 / 수신동의) 권장. 탈퇴가 복잡하면 2-A에서 멈추고 보고 후 2-B 별도 진행.

## WAVE 3 — 온보딩 체크리스트 (다운로드→설치→인증→첫 사용)
- 결제 후 처음 로그인한 회원이 뭘 해야 하는지 안내하는 **단계별 체크리스트** UI를 대시보드에 추가.
- 단계(제품 성격에 맞게): ① 앱 다운로드 → ② 설치 → ③ 라이선스 인증 → ④ 첫 사용. 각 단계 완료 시 체크 표시.
- **완료 판정 기준**: 가능한 것만 자동 판정하고(예: 다운로드 클릭 여부·라이선스 활성 여부 등 기존 데이터로 알 수 있는 것), 자동 판정이 어려운 단계는 **사용자가 수동 체크**하거나 안내 링크만 제공. ⚠️ 자동 판정을 위해 새 추적 데이터가 필요하면 임의로 만들지 말고 보고.
- 이미 온보딩을 마친(라이선스 활성·앱 사용 흔적 있는) 회원에겐 체크리스트를 접거나 숨김.
- 상태 저장이 필요하면(수동 체크 유지) profiles나 별도 컬럼 — 마이그레이션 필요 시 Steve 직접 적용 방식으로 파일 작성.
- ⚠️ 이 기능은 "있으면 좋은" 성격이라, 기존 데이터로 완료 판정이 불명확하면 **가장 단순한 버전**(수동 체크 + 안내 링크)으로 하고 과설계 금지.
검증 → 커밋 → **STOP & 보고.**
