# fix-stage3-revenue-admin

You are a senior Next.js 15 (App Router) / TypeScript / Supabase / Tailwind v4 engineer on the **CoreZent** web codebase (멀티 상품 SaaS 스토어). 3단계 "매출·관리자 운영"을 Wave별로 처리한다. **사전 점검(READ-ONLY audit) 불필요 — 바로 작업.** 단, 작업 중 애매하거나 위험한 판단이 필요하면 **멈추고 한국어로 보고**한다.

## ★ 확정 사실 (재확인 불요)
- **`orders.amount`(및 결제 금액 필드)는 cents 단위 = 실제 금액 × 100.** 확인됨: KRW 실주문 `amount=990114` → 표시 **₩9,901** (÷100). USD도 동일 관습(`99`=$0.99). 
- **즉 금액 표시 = amount ÷ 100.** 통화는 `currency` 필드(KRW/USD) 따름. 지금 매출·주문금액이 ₩0·₩1·₩990,114처럼 깨지는 건 이 ÷100(및 KRW 표기)이 없어서다.
- 상품 가격 표시는 이미 `src/lib/price.ts`의 `formatPrice`(원화 정수 입력)가 처리 중. **매출/주문 amount는 cents라 성격이 다름** — 이쪽은 ÷100 후 통화 포맷.

## 공통 STRICT RULES
- 코드는 네가 안다. 사용자는 코드 세부를 모르니, **막히거나 애매한 결정이 생기면 추측하지 말고 멈추고 한국어로 보고**하고 지시를 기다린다.
- **보편 해결 / 하드코딩 금지.** 금액 변환은 **공용 헬퍼 1개**로(예: `formatOrderAmount(amountCents, currency)` — cents÷100 + currency별 포맷). 곳곳에 `/100` 흩뿌리지 말 것. 기존 `lib/price.ts`와 역할 구분(가격=정수 입력 / 주문=cents)하되 파일은 같은 곳에 둬도 됨.
- **결제·라이선스·웹훅 로직 불변**: HMAC·멱등성·variant_id 매칭·라이선스 발급 미접촉. **금액은 "표시"만 바꾼다 — DB 저장값(cents)은 그대로 둔다.** 저장 단위를 바꾸지 말 것.
- **환불/취소(Wave 3)는 실제 돈이 움직이는 동작** — 반드시 확인 모달 + 명시적 재확인. 조용한 실행 금지.
- **다른 작업자/다른 CC 세션 미커밋 변경 미접촉.** `git add .` 금지(변경 파일만 개별 add). **Wave 중 push 금지**(최종에서만).
- **각 Wave 끝**: `npx tsc --noEmit` + `npm run build` 통과 → 변경 파일만 개별 커밋 → 멈추고 보고 → 승인 후 다음 Wave. **(bug-detective·code-guardian은 Wave마다 돌리지 말 것 — 최종에 한 번만.)**
- 각 보고: **①관찰 ②근거(파일:줄) ③공용 영향 범위 ④결정적 코드임.** 한국어로.

## WAVE 1 — 매출·주문금액 KRW화 (÷100 + 통화 포맷) ★선행·핵심
- 금액을 표시하는 모든 관리자/회원 경로에서 **amount(cents) ÷ 100 + currency별 포맷**을 적용. 공용 헬퍼 1개 신설해 전 지점이 공유(하드코딩 금지).
- 대상(전수 grep): 관리자 대시보드 매출 카드, 주문 목록/테이블 금액, 사용자 테이블 결제액, 회원 대시보드의 결제/주문 이력 금액, 그 외 amount를 그대로 렌더하는 곳.
- currency=KRW → "₩9,901"(천단위 콤마·소수 없음), USD → "$0.99". 혼재 데이터(과거 USD + 신규 KRW)를 currency 필드로 분기.
- ⚠️ **DB 저장값은 안 바꿈. 표시만.** 합계·통계 계산도 cents로 계산 후 마지막에 ÷100 표시(부동소수 오차 주의 — 정수 cents로 합산 후 표시 단계에서만 변환).
- 환불/부분환불 금액 필드가 별도로 있으면 동일 규칙 적용.
검증 → 변경 파일만 커밋 → **STOP & 보고.**
