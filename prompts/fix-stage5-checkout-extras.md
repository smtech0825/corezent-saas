# fix-stage5-checkout-extras

You are a senior Next.js 15 (App Router) / TypeScript / Supabase / Tailwind v4 engineer on the **CoreZent** web codebase (멀티 상품 SaaS 스토어, 결제=Lemon Squeezy). 5단계 "결제 확장" 3종을 Wave별로 처리한다. **사전 점검(READ-ONLY audit) 불필요 — 바로 작업.** 단, 작업 중 애매하거나 위험한 판단이 필요하면 **멈추고 한국어로 보고**한다.

## 공통 STRICT RULES
- 코드는 네가 안다. 사용자는 코드 세부를 모르니, **막히거나 애매한 결정이 생기면 추측하지 말고 멈추고 한국어로 보고**하고 지시를 기다린다.
- **보편 해결 / 하드코딩 금지.** in-place 수정, 지정 범위만, 무관 리팩터 금지.
- **결제·라이선스 핵심 불변**: HMAC 검증·order_id 멱등성·variant_id 매칭·slug 기반 분기({family}_{tier}_{interval})·라이선스 발급 로직 미접촉. 이번 작업은 **체크아웃 진입 전(수량·쿠폰 전달)과 알림(메일)** 계층.
- **금액 표시**: orders.amount는 cents(÷100) — 3단계에서 만든 공용 헬퍼 재사용. 새 포맷 함수 금지.
- **다른 작업자/다른 CC 세션 미커밋 변경 미접촉.** `git add .` 금지(변경 파일만 개별 add). **Wave 중 push 금지**(최종에서만).
- **각 Wave 끝**: `npx tsc --noEmit` + `npm run build` 통과 → 변경 파일만 개별 커밋 → 멈추고 보고 → 승인 후 다음 Wave. **(bug-detective·code-guardian은 Wave마다 돌리지 말 것 — 최종에 한 번만.)**
- 각 보고: **①관찰 ②근거(파일:줄) ③공용 영향 범위 ④결정적 코드임.** 한국어로.
- 스키마 변경이 필요하면 **마이그레이션 파일만 작성**, 적용은 Steve가 SQL Editor에서 직접(CC는 DB 직접 적용 금지).

## WAVE 1 — 수량(좌석) 선택 (장바구니 아님 — 같은 상품 N개)
- **범위 확정: 같은 상품을 수량만 늘려 결제.** 서로 다른 상품을 담는 장바구니는 이번 범위 아님(추후 별도).
- 가격 카드/구매 버튼 근처에 **수량 선택 UI**(1~N, 기본 1). 선택한 수량이 LS 체크아웃으로 전달되어 수량×단가로 결제되게.
- **LS 체크아웃에 수량 전달 방법을 먼저 확인**: 기존 checkout_url 방식에서 수량 파라미터(예: LS checkout의 quantity 지원)가 URL/API로 가능한지. 
  - **URL 파라미터로 가능하면** 그 방식(가장 단순).
  - **불가하고 LS API로 동적 checkout 생성이 필요하면** 범위가 커지므로 **구현 전 멈추고 보고**(방식·공수 제시).
- 수량 결제 시 웹훅 처리 확인: order_items의 quantity가 라이선스 발급에 어떻게 반영되는지(수량 3 → 라이선스 3개? 1개?). **현 웹훅이 수량을 무시한다면 그 사실을 보고**하고, 라이선스 N개 발급 로직 변경은 승인 후(결제·라이선스 핵심이므로 임의 변경 금지).
- ⚠️ 구독 상품의 수량(좌석) 개념은 LS 구독 quantity 지원 여부에 따름 — 불명확하면 보고. 일회성/구독 중 가능한 쪽부터.
검증 → 변경 파일만 커밋 → **STOP & 보고.**

## WAVE 2 — 쿠폰·할인코드
- **LS의 Discounts 기능 활용이 원칙**(자체 쿠폰 시스템 신설 금지 — 결제는 LS가 하므로 할인도 LS가 계산해야 금액 어긋남이 없음).
- 구현:
  - 체크아웃 진입 시 **할인코드 입력칸**(또는 URL로 코드 전달) — LS checkout이 discount 코드 파라미터를 받는 방식 확인 후 연결.
  - 관리자에는 **LS 할인코드 관리 안내/링크**(LS 대시보드에서 생성) 수준부터. LS API로 사이트 내 쿠폰 생성/관리까지 만들지는 **공수 확인 후 보고**(과설계 금지 — LS 대시보드로 충분하면 링크만).
- 웹훅의 주문 금액 처리에 할인 적용 금액(discount_total 등)이 오면 표시(3단계 주문 상세)에 반영 — 필드 존재 확인 후.
- ⚠️ LS discount 파라미터/필드가 코드·문서로 불명확하면 멈추고 보고.
검증 → 커밋 → **STOP & 보고.**

## WAVE 3 — 자동갱신 사전 알림 (갱신 전 메일)
- 구독 회원에게 **갱신(재결제) N일 전 안내 메일** 발송. 기존 Nodemailer 인프라 재사용.
- **발송 트리거 방식을 먼저 결정·보고 후 구현** (서버리스 환경이라 상시 크론이 없을 수 있음):
  - (a) **LS 웹훅 이벤트 활용**: LS가 갱신 임박/결제 예정 이벤트(subscription_payment_* 등)를 보내면 그 시점 활용 — 지원 이벤트 확인.
  - (b) **Vercel Cron**: 매일 1회 subscriptions에서 renews_at(갱신일)이 N일 뒤인 구독을 조회해 발송. Vercel cron 설정 필요.
  - 둘 중 현 구조에서 확실히 동작하는 쪽 선택. LS가 자체적으로 갱신 알림 메일을 보내는 설정이 있다면 그것도 확인·보고(자체 구현 불필요할 수 있음 — 가장 단순한 해결 우선).
- 중복 발송 방지(같은 갱신 건에 1회) — 발송 기록(컬럼 또는 로그 테이블, 3단계 Wave 7 이메일 로그 있으면 재사용).
- 수신 동의(2단계에서 만든 알림 동의)가 있으면 존중 — 단 **갱신 알림은 거래 알림 성격**이라 마케팅 동의와 별개로 발송하는 게 맞는지 판단 후 보고.
- 메일 내용: 상품명·갱신일·금액(cents÷100)·취소 안내 링크. 발신은 기존 corezent.com 메일 설정.
검증 → 커밋 → **STOP & 보고.**
